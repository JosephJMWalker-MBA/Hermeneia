# ADR-0019: Question Is Not a Subtype of Dialogue — FieldQuestion Is a Separate Class

**Status:** RATIFIED  
**Version:** 1.0  
**Date:** 2026-06-18  
**Supersedes:** None  
**Closes:** Q-P1-006 in EPISTEMIC_BACKLOG.md  
**Constitutional Cost of Error:** Medium

---

## Context

`question.py` exists as a class stub. `Question` appears in the field spec. `Dialogue` is defined in ADR-0017 as a human-authored object. The question is: can human-authored questions be Dialogue entries, and if so, what happens to system-generated questions (from coverage analysis, Perspective Debt surfacing)?

---

## Decision

**Human-authored questions are Dialogue entries (`dialogue_type: question`).**  
**System-generated questions are a separate canonical object: `FieldQuestion`.**

These must be distinguishable in the database without reading their text content. They are distinguishable by their object type: `Dialogue` vs. `FieldQuestion`.

---

## Formal Distinction

| Property | Dialogue (type: question) | FieldQuestion |
|---|---|---|
| Author | Human steward | System (compiler, planner, field analysis) |
| Origin | Human epistemic act | Algorithmic analysis of coverage or Perspective gaps |
| Provenance | Human provenance (steward ID) | Compiler provenance (pipeline ID, algorithm) |
| Purpose | Human curiosity, challenge, or inquiry | Prompting steward attention to a gap in the Field |
| Constitutional status | Human contribution (ADR-0017, ADR-0010) | System output — requires steward action to enter the dialogue layer |

---

## The Distinction Is Constitutional, Not Stylistic

ADR-0010 reserved certain decisions as human-only. The act of introducing an epistemic question into the Hermeneutic Field as a genuine human inquiry is constitutionally a human act.

If a `FieldQuestion` generated by the system were classified as a Dialogue entry, the system would have created a human epistemic contribution without human authorship. This is constitutionally prohibited.

The distinction is enforced by database table, not by a `type` flag on a shared table. `Dialogue` and `FieldQuestion` are separate tables with separate schemas. No join or union operation on these tables should produce a result implying they are the same kind of thing.

---

## The `question.py` Stub

The `question.py` stub is resolved as follows:
- It is refactored into `FieldQuestion`, representing system-generated questions only.
- Human-authored questions are handled by the existing `Dialogue` class with `dialogue_type: question`.
- The old `question.py` class stub is removed and replaced with `field_question.py`.

---

## FieldQuestion Schema

| Field | Type | Requirement |
|---|---|---|
| `id` | String (deterministic hash) | Required |
| `generating_system` | String | Required. Which system or pipeline generated this question (e.g. `perspective_debt_analyzer_v1`, `coverage_engine_v1`). |
| `generation_method` | Enum | Required. `coverage_gap`, `perspective_debt`, `contradiction_flag`, `other`. |
| `target_object_id` | String | Required. The object in the field that this question is about. |
| `target_object_type` | Enum | Required. Same enum as Dialogue's `target_object_type`. |
| `content` | String | Required. The question text, generated by the system. |
| `priority` | Enum | Required. `critical`, `high`, `medium`, `low`. Assigned by the generating system. |
| `status` | Enum | Required. `open`, `assigned`, `answered_by_dialogue`, `dismissed`. |
| `generated_at` | ISO 8601 datetime | Required. |
| `answered_by_dialogue_id` | String or null | The Dialogue entry ID that the steward authored in response to this FieldQuestion, if any. |
| `dismissed_by` | String or null | Steward ID who dismissed this question. |
| `dismissed_at` | ISO 8601 datetime or null | When it was dismissed. |
| `dismissed_rationale` | String or null | Why it was dismissed. |

---

## Lifecycle

A FieldQuestion is created by the system, reviewed by a steward, and either:
1. **Answered**: The steward authors a Dialogue entry addressing the question. The FieldQuestion's `answered_by_dialogue_id` field is updated. Status → `answered_by_dialogue`.
2. **Dismissed**: The steward decides the question is not worth pursuing. The `dismissed_by`, `dismissed_at`, and `dismissed_rationale` fields are populated. Status → `dismissed`.
3. **Remains open**: The steward has not yet acted on it.

A FieldQuestion that is answered by a Dialogue entry does not become that Dialogue entry. The two remain separate objects. The FieldQuestion persists as a record of what the system identified as a gap. The Dialogue entry is the human's response. They are linked by `answered_by_dialogue_id`.

---

## Validation Rules

```python
# FieldQuestions must be generated by the system, not human-authored
assert field_question.generating_system is not None
assert not is_human_steward(field_question.generating_system)

# Dialogue entries authored by humans may carry dialogue_type: question
assert dialogue.dialogue_type in VALID_DIALOGUE_TYPES
# "question" is a valid dialogue_type for human-authored entries

# FieldQuestions and Dialogue entries are distinguishable by object type alone
assert type(obj) in (Dialogue, FieldQuestion)  # never ambiguous
```

---

## Constitutional Alignment

- **ADR-0010** (Human-only decisions): Human epistemic questions are human contributions. System-surfaced questions are system outputs. The two cannot be conflated.
- **ADR-0017** (Dialogue definition): Dialogue is always human-authored. Adding a `dialogue_type: question` value extends the existing Dialogue schema without creating a new class.

---

## Consequences

**Positive:**
- The `dialogue_type: question` distinction within Dialogue preserves the full richness of human questioning (a steward asking "why does Fitzgerald use this word here?") without making it indistinguishable from a system-surfaced gap.
- FieldQuestions create a queryable queue of system-identified epistemic gaps for steward attention.
- The `answered_by_dialogue_id` link allows reconstruction of the conversation: "the system noticed this gap; the steward responded with this Dialogue entry."

**Negative:**
- Two tables for questions instead of one. The query "show me all open questions about Observation X" must union `FieldQuestions WHERE status=open AND target_object_id=X` with `Dialogue WHERE dialogue_type=question AND target_object_id=X`. This is manageable but adds query complexity.

---

## Alternatives Considered

**Alternative: Question as a subtype of Dialogue with `authored_by` null for system-generated entries**  
Rejected. A Dialogue entry with `authored_by: null` would violate the human provenance requirement on Dialogue (ADR-0017). Null is not a steward. A null author means "no human contributed this" — which is precisely what FieldQuestion is trying to represent. The solution is a separate object, not a nullable foreign key.
