# ADR-0009: AI Provenance Policy

**Status:** RATIFIED  
**Version:** 1.0  
**Date:** 2026-06-18  
**Supersedes:** None  
**Closes:** Q-P5-003 in EPISTEMIC_BACKLOG.md  
**Constitutional Cost of Error:** High

---

## Context

Hermeneia's Architect and Artist agents, the Perspective Generator, and the Transformation Planner all use language models to produce epistemic objects: Interpretations, NarrativeBlueprints, Perspectives, Concept graphs, and more. These objects may eventually be treated as canonical knowledge representations.

If a consumer of the system cannot determine which objects were produced by a language model, which model produced them, when, and under what circumstances, then the provenance chain is broken at the AI-to-human handoff. This is provenance debt of the highest order: it makes it impossible to audit the system's outputs for bias, error, model version drift, or unauthorized generation.

This ADR closes Q-P5-003 from `EPISTEMIC_BACKLOG.md`.

---

## Decision

**Every object generated by an AI system must carry an AI Provenance record. No exceptions.**

AI Provenance is not optional metadata. It is not a "nice to have" for debugging. It is constitutional infrastructure, required by Article II of the Constitution (Provenance mandatory) and by the principle that every epistemic object must be traceable to its origin.

"No AI-generated object becomes canonical merely because it exists. Its origin must remain permanently inspectable."

---

## Formal Definition

**AI Provenance** is a permanent, immutable record attached to an AI-generated object that identifies the generating model, the prompt or transformation that caused the generation, the parent objects it was derived from, and the human steward who accepted it into the canonical corpus.

AI Provenance does not validate the quality or correctness of the generated object. It records origin. Validity assessment is the responsibility of the steward who accepts the object.

---

## Required AI Provenance Fields

Every AI-generated object must carry a linked AI Provenance record containing all of the following:

| Field | Type | Requirement |
|---|---|---|
| `generating_model` | String | The canonical model identifier (e.g. `claude-sonnet-4-6`, `gpt-4o-2024-11-20`) |
| `model_version` | String | The specific model version used, if distinguishable from the identifier |
| `generation_timestamp` | ISO 8601 datetime | When the object was generated |
| `prompt_reference` | String or Object | A reference to the prompt template, transformation step, or agent instruction that caused the generation. Must be sufficient to reproduce the prompt context. |
| `parent_objects` | List[String] | IDs of the Hermeneia objects that were provided as input to the generation (e.g. Observation IDs, Concept IDs, ContinuityNode IDs). May be empty if no Hermeneia objects were inputs. |
| `generation_parameters` | Object | Model generation parameters at the time of generation (temperature, top_p, max_tokens, etc.) where available. |
| `accepting_steward` | String | Steward ID of the human who reviewed and accepted this object into the canonical corpus. |
| `acceptance_timestamp` | ISO 8601 datetime | When the steward accepted this object. |
| `acceptance_rationale` | String | Optional but encouraged. Brief statement of why the steward accepted this object. |

---

## Scope: Which Objects Require AI Provenance

AI Provenance is required for any object in `hermeneia.db` that was produced in whole or in part by a language model or any other AI system. This includes but is not limited to:

- Interpretations generated by the Architect agent
- NarrativeBlueprints generated by the Architect agent
- Prose renders generated by the Artist agent (if stored as canonical objects)
- Perspectives proposed by the Perspective Generator
- Concept graphs generated by any AI pipeline
- ContinuityNode proposals generated by entity recognition systems
- Relationship proposals generated by any AI pipeline
- Transformation plans generated by the Transformation Planner

Objects that are produced entirely by deterministic, non-AI pipelines (such as Observations produced by the Observation Compiler using rule-based sentence splitting) do not require AI Provenance. They require standard compiler Provenance as specified in ADR-0006.

---

## The Staging Protocol

No AI-generated object may be inserted directly into the canonical tables of `hermeneia.db`. All AI-generated objects must pass through a staging layer:

1. The AI system writes its output to a staging table (e.g. `proposed_interpretations`, `proposed_perspectives`).
2. The staging record carries the AI Provenance fields at the time of writing.
3. A human steward reviews the staged object.
4. The steward either:
   a. **Accepts** the object: it is moved from the staging table to the canonical table, with the steward's ID, acceptance timestamp, and rationale written to the AI Provenance record.
   b. **Rejects** the object: it is marked `rejected` in the staging table with the rejection rationale. It is never written to canonical tables. Rejected objects are never deleted — they remain in the staging table as a permanent record of what was generated and why it was not accepted.

The AI Provenance record is written at generation time (step 1) and completed at acceptance time (step 4a). It is a single record that spans both events.

---

## What AI Provenance Is Not

AI Provenance does not imply that the generating model is correct, reliable, or authoritative. An Interpretation accepted from an AI model carries exactly the same epistemic weight as its provenance would warrant — no more, no less.

AI Provenance does not exempt an object from quality review. Steward acceptance is required precisely because AI-generated objects must be evaluated by a human before entering the canonical corpus.

AI Provenance does not replace steward provenance. An accepted AI-generated object carries both its AI Provenance record (origin: which model, which prompt) and its steward provenance record (acceptance: which human, when, why). These are separate records serving separate auditing purposes.

---

## Inclusion Criteria

The following require AI Provenance:

1. Any Hermeneia object whose `text`, `content`, or structural fields were produced by a language model inference call.
2. Any Hermeneia object that was produced by a pipeline that includes a language model, even if the language model's output was subsequently post-processed.
3. Any ContinuityNode proposal generated by an entity recognition system that uses a neural model.
4. Any Relationship proposed by an AI system, even if the endpoints (ContinuityNodes) were created by humans.
5. Any clustering, grouping, or categorization of objects produced by an AI system.

---

## Exclusion Criteria

The following do not require AI Provenance:

1. Observations produced by the Observation Compiler using deterministic rule-based extraction (no language model involved). These require compiler Provenance per ADR-0006.
2. The Manifest (storage artifact, per ADR-0008).
3. ContinuityNodes created by human stewards without AI assistance.
4. Steward-authored annotations, rationales, or notes.

---

## Edge Cases

**An Interpretation produced by a pipeline that uses both rule-based and AI components**  
→ Requires AI Provenance. If any component of the pipeline involves a language model, the output carries AI Provenance.

**An AI-generated object that was substantially edited by a human steward before acceptance**  
→ Still requires AI Provenance, recording the original generation. The steward's edits are recorded in the acceptance_rationale field and in a separate steward revision record. The distinction between "AI-generated with steward edits" and "steward-authored" must be preserved.

**A prompt that itself contains Hermeneia Observations**  
→ The `parent_objects` field must list the Observation IDs provided as input to the prompt. This allows the provenance chain to flow from the source document through the Observation through the prompt through the AI-generated output.

**A model that does not expose its version identifier**  
→ The `generating_model` and `model_version` fields are populated with the best available information (e.g. the API endpoint used). If the model is genuinely not identifiable, a `model_unknown: true` flag must be set in the AI Provenance record and the generating_model field must contain the API endpoint or service name.

---

## Serialization Rules

AI Provenance is stored as a row in the `ai_provenance` table in `hermeneia.db`. The `ai_provenance` table is write-once, append-only. No UPDATE or DELETE on `ai_provenance` rows is permitted.

The `prompt_reference` field may be:
- A string reference to a prompt template in the `prompt_templates` table (for structured, versioned prompts).
- A SHA-256 hash of the prompt text (for ad hoc prompts).
- A structured object recording the prompt in full (for one-off or experimental generations).

The choice is recorded in a `prompt_reference_type` field: `template_id`, `hash`, or `full_text`.

---

## Validation Rules

```python
assert ai_provenance.generating_model is not None
assert ai_provenance.generation_timestamp is not None
assert ai_provenance.prompt_reference is not None
assert ai_provenance.accepting_steward is not None
assert ai_provenance.acceptance_timestamp is not None
assert ai_provenance.acceptance_timestamp >= ai_provenance.generation_timestamp
# No AI-generated object may exist in canonical tables without a linked ai_provenance record
assert ai_provenance_exists_for(canonical_object.id)
```

Additionally:

- The `ai_provenance` table must have a foreign key constraint linking each record to the canonical object it describes.
- The staging tables must enforce that no record moves to canonical tables without `accepting_steward` and `acceptance_timestamp` populated.

---

## Migration Policy

1. AI Provenance records created under ADR-0009 are permanent. They are never deleted.
2. If this definition is amended (new required fields), existing AI Provenance records are not retroactively required to carry the new fields. A `schema_version` field in the `ai_provenance` table identifies which version of this ADR governed the record.
3. New AI Provenance records created after an amendment must carry all fields required by the amended definition.
4. If the `generating_model` identifier scheme changes (e.g. model names are restructured), existing records retain the identifier used at generation time. An alias table may be maintained for cross-referencing.

---

## Constitutional Alignment

- **Article II** (Provenance mandatory): AI Provenance is a direct application of the constitutional provenance requirement to AI-generated objects. The provenance chain must be traceable from source document → Observation → AI input → AI output → steward acceptance → canonical object.
- **Article VI** (if applicable): Any article governing the role of AI in the system is directly satisfied by this ADR.
- **Axiom 6** (Provenance mandatory): The axiom applies without exception. AI generation is not an exception.
- **ADR-0010** (Human-only decisions): The steward acceptance step in the staging protocol is the implementation of the human-only decision requirement for AI-generated objects.

---

## Consequences

**Positive:**
- Every AI-generated object in the canonical corpus is permanently traceable to its origin model, prompt, and accepting steward.
- Model drift can be audited: if a model is deprecated or found to have systematic biases, all objects it generated can be identified and reviewed.
- The boundary between AI-generated and human-generated content is always inspectable.
- The staging protocol prevents unreviewed AI output from entering the canonical corpus.

**Negative:**
- The staging protocol adds latency to AI-assisted workflows. Human steward review is a bottleneck.
- Prompt storage adds database overhead, especially for ad hoc prompts stored as full text.
- If the `generating_model` field is not precisely specified by the API (some models do not expose version-level identifiers), provenance records may be less precise than desired.

---

## Alternatives Considered

**Alternative: AI provenance optional, included at steward discretion**  
Rejected. "Optional" provenance creates a two-tier epistemic world in which some objects are auditable and others are not. There is no principled reason to exempt any AI-generated object from origin traceability. The constitutional mandate of Article II admits no exceptions.

**Alternative: Embed AI provenance as fields on each canonical object rather than a separate table**  
Considered. Rejected in favor of a separate `ai_provenance` table because: (1) it cleanly separates epistemic content from generation metadata; (2) it allows a single AI generation event to produce multiple canonical objects with one provenance record; (3) it allows the AI provenance schema to evolve independently of the canonical object schemas; (4) it mirrors the separation used for compiler Provenance in ADR-0006.
